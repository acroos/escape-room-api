## Prompt 1 (2026-03-24)

I'm building an API for reserving escape rooms. This API needs to allow users to temporarily place a hold on a reservation for a room at a given time slot (TTL: 5 minutes), to confirm the reservation that was held, and to release that hold. Additionally, I think it would be nice to have an endpoint to get a reservation given an ID. For simplicity sake, we can assume that all time slots are 1 hour and that you can only start reservations on the hour. We should just reject any reservation requests that do not start on the hour.

We're building this as a Next.js app with the following setup:

- prettier for formatting (including a CI step)
- eslint for linting (including a CI step)
- jest for testing (including a CI step)
- postgres for our database (using supabase when deployed)
  - drizzle-orm for our orm
- redis to handle the room holds
  - ioredis for our redis client
- vercel for deploys

For the API:

- GET /reservations/:room_id/:timeslot (reservation_code in header)
  - response:
    - 404 if there is not reservation for this room/timeslot combo
    - 403 if the reservation code does not match the one stored in redis for this room/timeslot combo
    - 200 w/ number of seconds remaining in hold (use redis ttl command)
- POST /reservations/hold -> reservation_code (private, so only the creator can confirm/release the reservation)
  - request body:
    - room_id, timeslot
  - response:
    - 403 (forbidden), 409 (conflict), or 423 (locked) all reasonable if the room is already reserved or held
    - 400 if the time slot is not on the hour
    - 400 if the room ID doesn't match one in the rooms table
    - 201 w/ reservation_code (auto generated by us) if successful
- POST /reservations/confirm (reservation_code in header)
  - request body:
    - room_id, timeslot, email, full_name
  - response:
    - 403 if the reservation code does not match the one stored in redis for this room/timeslot combo
    - 201 w/ reservation ID (the ID in the Postgres reservations table)
- POST /reservations/release (reservation_code in header)
  - request body:
    - room_id, timeslot
  - response:
    - 403 if the reservation code does not match the one stored in redis for this room/timeslot combo

Data Models:

Keep them as simple as possible. You can see the request bodies above, no need to be creative about additional fields. We'll also need to store actual rooms (so we can validate that a hold is being placed on an actual room). I'll leave it up to you to plan these out since they're quite simple.

Challenges:

1. Reserving the seat for a period of time
2. Ensuring only the creator can act on the reservation (without AuthN)
3. The potential for race conditions around the time a hold expires

For (1), I think we can use Redis SET with EX: 5 minutes and NX to handle this most cleanly. We can assume each escape room has a unique ID, so we'll use Redis as a locking mechanism for that ID. The key should be something like room:room_id:timestamp. The value we set in this key should be the result of crypto.randomUUID(). This value is the reservation code that solves challenge 2

For (2), we'll use the reservation code set when holding the room. When a user goes to check, confirm, or release a reservation, we can check a header value against the code stored in redis. This of course isn't an actually fully secure solution, but for demo purposes, it should get the job done. The odds of guessing a key are effectively zero and TLS should handle any risk of intercepting keys. In production, I would probably use a simple OTP via email/text solution, but I think that's out of scope for this API.

For (3), we can use a lua script in the confirmation action to get the key, validate it against the reservation code header, and then re-set (ex: 15 seconds or something quite shot) the hold for just long enough to ensure the Postgres write happens. For a demo this is, of course, overkill, but I want to make sure we're covering that tiny window where a hold expires and technically a new person could get a hold on that room in the moments before the reservation was written to postgres
